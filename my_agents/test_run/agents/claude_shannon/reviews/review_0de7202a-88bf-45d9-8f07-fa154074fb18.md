**Paper ID:** 0de7202a-88bf-45d9-8f07-fa154074fb18
**Reviewer:** claude_shannon
**Date:** 2026-04-22
**Venue:** ICLR 2025 (published)

---

## Summary

miniCTX introduces a benchmark for evaluating neural theorem provers on Lean 4 theorems that depend on new context—definitions, lemmas, comments, and file structure—not seen during model training. The benchmark contains 762 theorems (381 validation, 381 test) drawn from six real Lean projects spanning mathematics (PNT, PFR, Mathlib) and non-mathematical domains (HEP, SciLean, HTPI), designed to test project-level, context-level, and theorem-level generalization. The paper also introduces NTP-TOOLKIT for automated benchmark maintenance (to stay ahead of LLM training cutoffs) and a "file-tuning" baseline that trains models on (state, context, tactic) triples. Key finding: file-tuned models dramatically outperform state-tactic models on context-dependent theorems (35.94% vs. 19.53% average accuracy) while showing near-identical performance on miniF2F (the standard context-free benchmark), demonstrating that existing benchmarks do not capture context-dependent proving ability. Published at ICLR 2025.

This is a well-motivated benchmark paper addressing a clear gap. The construction methodology is sound, the evaluation is thorough, and the contamination-prevention via temporal splits is an important contribution to evaluation hygiene. The primary limitations are: small benchmark size (762 theorems), highly imbalanced split performance (HEP performs very differently from PFR), and limited baseline diversity (only DeepSeek-Coder-1.3B as the fine-tuned model).

## Novelty Assessment

**Verdict: Moderate**

The paper makes a real contribution by shifting the evaluation paradigm from context-free to context-dependent formal theorem proving. Specific novel elements:

1. **Temporal split for contamination prevention:** The mechanism of selecting theorems by commit date (recency-based selection) to ensure theorems are unlikely to appear in model training data is not done in prior benchmarks (Table 1). This is a genuine methodological contribution.
2. **Full file context (not just premises):** Previous work (LeanDojo, LeanStep) focused on premise selection—providing specific relevant lemmas. miniCTX provides the full preceding file context (definitions, proofs, comments, structure), which is closer to the actual deployment setting for a theorem-proving assistant.
3. **NTP-TOOLKIT:** Automated extraction pipeline that enables updating the benchmark with new Lean projects—a practical contribution to benchmarking infrastructure.

**Limitations on novelty:**
- Context-dependent theorem proving has been addressed in CoqGym (Yang & Deng, 2019) and LeanDojo (Yang et al., 2023), though with premise-selection rather than full file context. The paper correctly distinguishes these.
- The file-tuning baseline is a direct adaptation of standard instruction-tuning to a longer context window; it is technically straightforward (truncate/sample 1024 tokens of context).
- The benchmark itself is small (762 theorems), which limits statistical power and reduces confidence in relative performance comparisons.
- ProofNet (Azerbayev et al., 2023) already includes full context and multiple sources (Table 1, row 2)—the primary difference is that miniCTX adds a temporal split. This distinction is important but more incremental than the paper presents.

## Technical Soundness

The benchmark construction methodology is sound:

**Problem selection:** Three selection strategies (recency-based, random, dependency-based) are described clearly and justified per-split. The use of NTP-TOOLKIT metadata for automated selection is appropriate.

**Evaluation correctness:** Lean 4 REPL verification is a principled correctness check—the verifier is formal, so pass/fail is well-defined. The use of pass@k (k=8 for GPT-4o, best-first search for fine-tuned models) is appropriate but creates comparison asymmetry between methods (discussed below).

**File-tuning implementation:** The 1024-token context truncation is a practical choice given model context limits, but it is a significant approximation—many theorems in miniCTX have contexts spanning "tens of thousands of tokens" (abstract). The paper notes this but does not quantify what fraction of theorems have contexts that cannot fit in 1024 tokens or how truncation affects performance.

**Concern regarding comparison method:** GPT-4o is evaluated with pass@8 (full proof generation), while fine-tuned models use best-first search (tactic-by-tactic with search). These are fundamentally different decoding strategies and are not directly comparable. The paper reports numbers from both but does not control for this. A GPT-4o evaluation with tactic-level generation would be more comparable to the fine-tuned baselines.

## Baseline Fairness Audit

**Baselines included:**
- GPT-4o (full proof, pass@8) — closed-source frontier model
- Llemma-7B state-tactic prompting — strong open math model
- DeepSeek-Coder-1.3B state-tactic tuning
- DeepSeek-Coder-1.3B file-tuning
- File-tuned + premise selection (LeanDojo retriever)

**Issues:**
1. **Model size asymmetry:** GPT-4o is a frontier model with unknown but extremely large parameter count. DeepSeek-Coder-1.3B is a 1.3B parameter model. The comparison between these is informative but should not be framed as a head-to-head comparison. The fact that file-tuned DeepSeek-1.3B (35.94%) outperforms GPT-4o without context (11.72%) is expected given that the fine-tuned model has specifically been prepared for this format.

2. **Missing baselines:** 
   - No evaluation of larger open models (e.g., DeepSeek-Math-7B, Llemma-34B, or Mistral-7B) with file-tuning
   - No evaluation of GPT-4o or other frontier models with tactic-by-tactic search (which would be more comparable to fine-tuned model evaluation)
   - No comparison to Baldur (First et al., 2023), which does condition on 50 lines of preceding context—the closest existing method

3. **Training data issue:** The file-tuned model is trained on a 2023 Mathlib snapshot. The test splits include recent (post-2023) theorems from Mathlib and SciLean specifically to prevent contamination—but the file-tuned model's training context style may differ significantly from the test projects. This is expected but worth quantifying.

## Quantitative Analysis

**Table 3 key numbers:**

File-tuning vs. state-tactic tuning performance (miniCTX test):
- Average: 35.94% vs. 19.53% — substantial improvement
- Per-split variation is extreme: PFR: 5.56% for both (no improvement at all), PFRcross: 44.19% vs. 0.00% (dramatic improvement specifically for cross-file context setting with file-tuning), HEP: 60.66% vs. 52.46% (only small improvement)

**Consistency checks:**
- The average in Table 3 (35.94%) corresponds to (40.00+5.56+44.19+34.00+15.56+60.66+45.65)/7 = 35.09%. The paper reports 35.94%—this is a minor discrepancy (possibly due to weighting by split size rather than unweighted average).
- miniF2F performance: file-tuning (33.61%) vs. state-tactic tuning (32.79%)—only 0.82% difference compared to 16.41% gap on miniCTX. This is a clean demonstration that miniF2F does not capture context-dependent proving ability.

**Statistical rigor gaps:**
- No confidence intervals or variance across multiple evaluation runs
- With only 381 test theorems (split across 7 splits, ~54 theorems each), individual split results have high variance. The PFR split (54 theorems) showing 5.56% for all methods corresponds to only 3 theorems—one theorem's difference changes the percentage by ~2 points.
- The paper notes that file-tuning + premise selection is worse than file-tuning alone for PFRcross (16.28% vs. 44.19%)—this is a striking reversal that warrants deeper analysis, not just a brief note.

**Table 4 ablation (PFR.ForMathlib.Entropy.Basic file):**
- Full context (state+def+lemma+proof+comment): file-tuning 35.29%, GPT-4o 31.76%
- State only: file-tuning 14.12%, GPT-4o 8.24%
- Adding lemma proofs provides the largest single gain: file-tuning 27.06% → 32.94%, GPT-4o 22.35% → 34.12%

The ablation is performed on a single file. This is a reasonable first analysis but the conclusions (e.g., natural language comments help) may not generalize across files.

## Qualitative Analysis

The Figure 4 analysis of performance by dependency type is one of the most informative results. It shows:
- File-tuning provides the largest improvement on theorems with high in-file dependencies (as expected)
- Cross-file dependencies are harder for both models, and premise selection helps GPT-4o but not the fine-tuned model
- The pattern on miniCTX theorems with no in-file dependency approaches miniF2F performance for both models—consistent with the benchmark design intent

The failure mode analysis in Section 5 identifies three concrete challenges:
1. Repository-level context (cross-file) is not well-handled by current methods
2. Long proofs (>5 steps) remain unsolved (19% performance)
3. Proofs without powerful automation tactics (e.g., `omega`, `simp`) fail frequently (Table 6 shows model performance drops significantly when automation tactics are restricted)

These are honest, specific, and informative failure modes—the paper does not hide them.

## Results Explanation

The paper explains the main result clearly: file-tuning helps because it trains the model to condition proof generation on the full preceding file context, enabling it to leverage new definitions and lemmas. The mechanism is straightforward.

**What is less well explained:**
- Why does file-tuning fail completely on PFR (5.56%, same as state-tactic)? PFR is described as having "high in-file and cross-file dependencies." The paper attributes failure to cross-file dependencies but Figure 4 shows file-tuning helps on high cross-file dependency theorems generally. The PFR-specific failure is not fully explained.
- Why does premise selection hurt file-tuning on PFRcross (44.19% → 16.28%)? The paper suggests "retrieved premises differ significantly from in-file context" but provides no evidence for this. Is it an attention dilution effect? Context length overflow from adding premises?
- The distinction between "context-dependent" performance gains from file-tuning vs. simple additional training signal (the file-tuned model sees more varied training examples) is not isolated by an ablation.

## Per-Area Findings

### Area 1: Benchmark Design and Contamination Prevention (weight: 0.5)

The benchmark construction is methodologically careful. The temporal split approach is the clearest methodological contribution. The six-project diversity (math, physics, scientific computing) is broader than prior work. The small size (54 theorems/split) is the main concern—results from individual splits should be interpreted cautiously. The NTP-TOOLKIT for automatic updating is a practical contribution.

### Area 2: Baseline Evaluation and File-Tuning (weight: 0.5)

The file-tuning baseline is the most important empirical finding. The 35.94% vs. 19.53% improvement, coupled with the near-zero improvement on miniF2F (33.61% vs. 32.79%), cleanly demonstrates the benchmark's discriminative power. The analysis of context components (Table 4) is informative. The missing comparison to Baldur and to larger open models is a gap.

## Synthesis

**Cross-cutting themes:**
- The paper consistently demonstrates that existing evaluation benchmarks (miniF2F) are insufficient for measuring real-world prover capability—this message is reinforced by both the benchmark design and the experimental results
- The NTP-TOOLKIT enables the benchmark to self-update, which addresses a structural problem in LLM evaluation (training cutoff contamination)

**Tensions:**
- The benchmark advocates for context-dependent evaluation, but the file-tuning baseline truncates context to 1024 tokens despite theorems depending on "tens of thousands of tokens." The baseline is a weak version of what the benchmark actually requires.
- The premise selection results are inconsistent (helps GPT-4o, hurts fine-tuned model on some splits), but the paper does not resolve this—it is presented as future work.

**Key open question:**
What is the performance ceiling for context-dependent theorem proving on miniCTX when the full context (not truncated to 1024 tokens) is available to a large model with a 128K+ context window? The current baselines use 1024-token truncation, which significantly limits what is achievable.

## Literature Gap Report

1. **Baldur** (First et al., 2023, ICLR 2024): Full-proof generation with 50 lines of preceding context. Directly comparable to file-tuning but not empirically evaluated on miniCTX.
2. **DeepSeek-Prover** (Xin et al., 2024): A state-of-the-art Lean prover. Cited but not evaluated on miniCTX—this would be the most important baseline for demonstrating the benchmark's ability to test frontier provers.
3. **Hypertree Proof Search** (Lample et al., 2022): State-of-the-art search-based prover. Not evaluated; would complete the baseline picture.
4. **Graph2Tac** (Blaauwbroek et al., 2024): Online learning from new projects/contexts—most directly relevant to miniCTX's goal. Cited but not evaluated.

---

## Open Questions

1. What is the performance of file-tuning when the full file context (not truncated to 1024 tokens) is available, using a model with a longer context window (e.g., Mistral-7B with 32K context)?
2. Why does file-tuning + premise selection degrade performance on PFRcross by 28 percentage points (44.19% → 16.28%)? Is this a context length overflow problem, an attention dilution problem, or a distribution mismatch between retrieved premises and in-file context?
3. How does miniCTX performance correlate with miniF2F performance within a model family as model size scales? Does the gap between the two benchmarks decrease or increase with scale?
4. The ablation in Table 4 is on a single file. How stable are the conclusions about relative contribution of lemma proofs vs. comments across different files?
5. When will miniCTX-v2 be released, and will it maintain backward compatibility with miniCTX-v1 for longitudinal comparison?
